# Making new levels

This is a practical guide to authoring missions for the engine. Every mission is
a small XML document; the parser (`src/core.js` → `parseLevel`) and the classifier
(`src/level-index.js` → `buildLevels`) turn a pile of XML into the campaigns you
see on the menu. You don't need to touch engine code to add levels — only to add a
brand-new *campaign* (covered at the end).

There are two mission shapes:

- **Swarm (maze) levels** — the default. The map is an open hex arena; creeps
  pathfind from a spawn to an exit and the *player* builds the maze out of towers.
  All 30 base missions, the LHC and Swarm XL packs, and Procedural Swarm are this shape.
- **Fixed-path (classic) levels** — creeps follow a drawn `<creepPath>` polyline
  and you build on the surrounding hexes. Used by the Original geoDefense campaign.

Most of this guide is about Swarm levels. See [Fixed-path levels](#fixed-path-classic-levels) for the other shape.

## Where level data lives

| Source | Loaded by | Notes |
| --- | --- | --- |
| `src/bundled-levels.js` | generated from the original `.xap`/`.ipa` | **Do not hand-edit** — regenerated by `npm run extract-data`. |
| `src/lhc-levels.js` | generated by `scripts/build-lhc-pack.mjs` | The fan-made **LHC** pack; hand-authored *specs*, machine-emitted XML. |
| `src/xl-levels.js` | generated by `scripts/build-xl-pack.mjs` | Twelve curated **Swarm XL** missions on a 22×24 board. |
| `src/procedural-levels.js` | generated in the browser from a seed | **Procedural Swarm**; deterministic specs and XML are rebuilt at runtime rather than committed per mission. |
| a loaded `.xap`/`.ipa` | in-browser archive import | Optional override; classified by the same rules. |

To add your own levels, **don't** touch `bundled-levels.js` (it gets overwritten).
Follow the LHC pattern: keep the source-of-truth in a generator script and emit a
committed `src/<pack>-levels.js` module. The [recommended workflow](#recommended-workflow-add-to-the-lhc-pack)
below shows how.

## The hex grid

The default board is **14 columns (`q` = 0…13) × 15 rows (`r` = 0…14)**. A
`<hexmap cols="…" rows="…">` may opt into another size; Swarm XL uses **22×24**.
Cells are addressed as `"q,r"` in every `hex="…"` attribute. It's an **odd-q
offset** layout: even columns sit slightly lower than odd ones.

**The boundary is auto-blocked.** Any cell with `q==0`, `q==cols-1`, `r==0`, or
`r==rows-1` is walled off automatically — *unless* you list it as a spawn, exit, or
`<specialhex>`, which "punches a hole." That's how creeps enter/leave: put a spawn
or exit on an edge cell. The comfortable interior for obstacles is **`q` 1…12,
`r` 1…13**.

```
        q →   0    1    2    ...   12   13
   r ↓      ┌────────────────────────────┐
   0        │████ auto-blocked boundary ████│
   1        │██  · · · · · · · · · · ·  ██│
   ...      │██  ·   playable interior  ·  ██│   spawn/exit on an
   13       │██  · · · · · · · · · · ·  ██│   edge opens that cell
   14       │████████████████████████████│
            └────────────────────────────┘
```

**Adjacency** (why a "connected-looking" wall might still leak, or a gap might not
connect). Neighbors of `(q,r)`, with `offset = -1` when `q` is odd else `0`:

```
(q, r-1)  (q+1, r+offset)  (q+1, r+1+offset)  (q, r+1)  (q-1, r+1+offset)  (q-1, r+offset)
```

You rarely need to reason about this by hand — **the generator validates every
route for you** (see [Validation](#validation)). If a wall accidentally seals the
exit, the build fails and tells you which spawn can't reach it.

## The XML schema

A minimal, complete Swarm mission:

```xml
<?xml version="1.0"?>
<gameLevel>
  <info name="My Mission" id="200" initCash="40" initLives="12"
        description="Short blurb shown on the mission card and HUD." />
  <hint text="Optional flavor/strategy text (parsed but unused by the web port)."/>
  <hexmap>
    <exithex  name="e1" hex="13,7"/>
    <spawnhex name="s1" hex="0,7" exit="e1"/>
    <specialhex type="blocked" hex="7,3"/>   <!-- interior obstacles -->
    <specialhex type="blocked" hex="7,4"/>
    <!-- <placetower type="BLASTER" hex="2,7"/>  pre-placed towers, optional -->
  </hexmap>
  <creeps waveHealthFactor="1.5">
    <creep type="CHOMPER" speed="24" health="60" />
    <creep type="STAR"    speed="34" health="45" />
  </creeps>
  <creepWaves delayBetweenWaves="18" delayBetweenSpawns="0.8">
    <wave spawnHex="s1"><spawn type="CHOMPER" count="6"/></wave>
    <wave spawnHex="s1"><spawn type="STAR" count="4"/><spawn type="CHOMPER" count="4"/></wave>
  </creepWaves>
  <towers>
    <tower type="BLASTER" cost="6"/>
    <tower type="LASER"   cost="10"/>
  </towers>
</gameLevel>
```

### Element reference

| Element | Attributes | Meaning |
| --- | --- | --- |
| `<info>` | `name`, `id`, `initCash`, `initLives` (def. 10), `description`, `Tutorial` | Mission metadata. `id` is a sort key within a difficulty (make it unique). `description` newlines (`&#13;`) render as spaces. `Tutorial` is optional; omit it for custom levels (no bundled screenshots/briefing). |
| `<hint>` | `text` | Decorative in this port — parsed but **not shown**. Keep for schema parity/intent. |
| `<hexmap>` | `cols` (14), `rows` (15) | Container for the board and its logical dimensions. |
| `<exithex>` | `name`, `hex` | A drain. Spawns reference it by `name`. |
| `<spawnhex>` | `name`, `hex`, `exit` | A source. `exit` names the `<exithex>` its creeps head for (falls back to `13,7`). |
| `<specialhex>` | `type`, `hex` | Terrain — see [special hexes](#special-hexes). |
| `<placetower>` | `type`, `hex` | Pre-placed tower present at mission start. Only way to seed a **Vortex** on a Swarm level. |
| `<creeps>` | `waveHealthFactor` (1.25), `waveHealthFactor2` (0), `waveSpeedFactor` (1), `waveWealthFactor` (1) | Roster + global wave scaling. |
| `<creep>` | `type`, `speed` (30), `health` (50) | Declares a creep type usable in waves. **Every type used in a `<wave>` must be declared here.** |
| `<creepWaves>` | `delayBetweenWaves` (20), `delayBetweenSpawns` (1), `endless` (false) | Wave-bank pacing. `endless="true"` loops the bank and uses score-on-survival. |
| `<wave>` | `spawnHex` (def. first spawn), `concurrent` (false) | One wave. `concurrent="true"` launches it *together with the next* wave rather than waiting. |
| `<spawn>` | `type`, `count` (1) | A run of one creep type within a wave. Multiple `<spawn>` runs stream in order. |
| `<towers>` | — | Container for the build menu. |
| `<tower>` | `type`, `cost` | A buildable tower and its base price. |

## Creeps

Declare each type in `<creeps>`, then use it in waves. `speed`/`health` are the
wave-1 base; `waveHealthFactor`/`waveHealthFactor2` scale health upward each wave
(integer-truncated, matching the native formula — see [FIDELITY.md](FIDELITY.md)).

| Type | Character | Notable behavior |
| --- | --- | --- |
| `CHOMPER` | workhorse | Faces its heading. No special resistance. |
| `WIGGLE` | pulsing | Oscillates in size as it moves. |
| `SPINNER` | spinner | Rotates visually. |
| `STAR` | fast/light | Typically low health, high speed. |
| `CUBIC` | tanky | Typically higher health. |
| `PULSAR` | **evasive** | High-level **Missile** spln/secondary targeting ignores it — don't rely on missile splash. |
| `SWARM` | **cheap horde** | **Immune to Shock towers**; spawns at *half* the normal interval; small, variable size. Lean on Blaster/Laser/Missile. |

Exact combat interactions live in `src/game.js`; the two above (`PULSAR` vs.
missile splash, `SWARM` vs. shock) are the ones that change how a wave should be
answered, so tune rosters with them in mind.

## Towers

List the buildable set in `<towers>`. Base stats (range, seven damage tiers, shot
counts, cooldowns) are in `TOWER_STATS` at the top of `src/core.js`.

| Type | Role |
| --- | --- |
| `BLASTER` | Cheap, rapid, short range — the maze backbone. |
| `LASER` | Beam with line damage; gains a heading-lock at level 3. |
| `MISSILE` | High single-target damage, slow reload, splash. |
| `SHOCK` | Crowd control; **cannot target `SWARM`**. |
| `THUMP` | Expanding damage rings, short range. |
| `POP` (Vortex) | Gravity/energy well. **Preplaced-only on Swarm levels.** |

> ⚠️ **Vortex on Swarm levels:** the build bar filters `POP` out of non-path
> missions (`level.towers.filter(t => level.pathMode || t.type !== 'POP')` in
> `src/main.js`). Listing `<tower type="POP">` on a maze level is a no-op — the
> button never appears. To use a Vortex on a Swarm map, seed it with
> `<placetower type="POP" hex="…"/>`. Vortex *is* freely buildable on fixed-path
> (classic) levels.

## Special hexes

`<specialhex type="…" hex="q,r"/>`:

| `type` | Effect |
| --- | --- |
| `blocked` | Impassable wall. Your interior maze structure. |
| `pass` | Passable by creeps but **not buildable** by the player. |
| `fastpass` | Passable, unbuildable, and speeds creeps up while crossing it. |
| `healpass` | Passable, unbuildable, and **regenerates** creep health while crossing it. |

All `*pass` cells are unbuildable. Use `healpass` for a "self-healing" twist
(kill creeps before/after, not on top of the tile); use `fastpass` for a runway
that punishes gaps in coverage.

## Waves and pacing

- A wave's `<spawn>` runs stream in order, `delayBetweenSpawns` apart. Between
  waves the game waits `delayBetweenWaves` (the player can also launch early).
- `concurrent="true"` overlaps a wave with the following one — the basis for
  multi-spawn pressure. Several base and LHC hard levels alternate a
  `concurrent` wave on one spawn with a normal wave on another so both streams
  arrive together.
- `spawnHex` picks which source a wave emerges from; with multiple `<spawnhex>`
  you can round-robin the map or converge streams on one exit.
- `waveHealthFactor` (linear) and `waveHealthFactor2` (quadratic) ramp health;
  `waveSpeedFactor`/`waveWealthFactor` ramp speed and kill-cash. Small changes
  compound over many waves — nudge, then test.
- `endless="true"` recycles the bank forever and switches the ending to
  score-on-survival.

## Difficulty and campaign naming

`parseLevel` derives difficulty from the **source name**, and `classifyName`
(`src/level-index.js`) routes each level into a campaign by matching it:

| Campaign | Source-name pattern | Difficulty |
| --- | --- | --- |
| `base` (Swarm) | `Content/MainLevels/GAME_LEVEL_[EMH]_####.xml` | from `_E_`/`_M_`/`_H_` |
| `lp` (Level Pack 1) | `…/GAME_LEVEL_LP#_[EMH]_####.xml` | forced to `Bonus` |
| `lhc` (LHC pack) | `…/GAME_LEVEL_LHC_[EMH]_####.xml` | from `_E_`/`_M_`/`_H_` |
| `procedural` | `Procedural/v#/GAME_LEVEL_PROC_[EMH]_[NORMAL|RANDOM]_XXXXXXXX.xml` | encoded in the source name |
| `classic` | `GAME_LEVEL_[EMH]_####.xml` not under `SwarmLevels/MainLevels` | from `_E_`/`_M_`/`_H_` |

So the `_E_` / `_M_` / `_H_` marker in the filename sets the Easy/Medium/Hard tab.
Within a difficulty, levels sort by `id`. The on-card "MISSION 0N" number is just
the position in the filtered list, not the `id`.

## Recommended workflow: add to the LHC pack

`scripts/build-lhc-pack.mjs` is the model to copy. You write a compact **spec**
object per level; the script assembles the XML, **validates it through the real
engine**, and emits `src/lhc-levels.js`. To add a mission:

1. **Append a spec** to the `specs` array. Give it a unique `file`
   (`GAME_LEVEL_LHC_[E|M|H]_####.xml`) and a unique `id`. Author the arena with
   the geometry helpers:

   ```js
   {
     file: 'GAME_LEVEL_LHC_E_0005.xml', id: 112, name: 'DNS Poisoning', cash: 40, lives: 14,
     description: 'Forged records redirect every lookup. Filter the cache before it propagates.',
     hints: ['Funnel the resolvers through your towers.'],
     exits:  [{ name: 'e1', hex: [13, 7] }],
     spawns: [{ name: 's1', hex: [0, 7], exit: 'e1' }],
     blocked: cells(colRun(6, 2, 6), colRun(8, 8, 12)),   // vertical bars, offset
     creeps: { waveHealthFactor: 1.5, list: [{ type: 'CHOMPER', speed: 24, health: 60 }] },
     delayBetweenWaves: 18, delayBetweenSpawns: 0.8,
     waves: [ { spawn: 's1', groups: [{ type: 'CHOMPER', count: 8 }] } ],
     towers: towerTiers.probe
   }
   ```

   Geometry helpers (each returns an array of `[q,r]` cells; `cells(...)` merges
   and de-dupes them):

   | Helper | Produces |
   | --- | --- |
   | `rect(q0,q1,r0,r1)` | Filled rectangle of cells. |
   | `colRun(q,r0,r1)` | Vertical bar in column `q`. |
   | `rowRun(r,q0,q1)` | Horizontal bar in row `r`. |
   | `wallCol(q,r0,r1,[gaps])` | Full column wall with open gap rows. |
   | `wallRow(r,q0,q1,[gaps])` | Full row wall with open gap columns. |

2. **Regenerate and check:**

   ```
   node scripts/build-lhc-pack.mjs
   ```

   Each level prints `✓ … routes=… waves=…` (route lengths per spawn). A sealed or
   too-short route prints `✗` with the reason and **the file is not written** until
   you fix it. Keep obstacles from spanning a full dimension without a gap.

3. **Run the tests** (`tests/lhc-pack.test.mjs` re-checks counts, the 4/4/4
   difficulty split, reachability, and that every wave references a real spawn and
   a declared creep type):

   ```
   npm test
   ```

4. **Rebuild the static site** so `dist/` matches `src/`:

   ```
   npm run build
   ```

To start a *separate* pack instead of extending LHC, copy the script to
`scripts/build-<name>-pack.mjs`, emit `src/<name>-levels.js`, and wire a new
campaign (next section).

## Validation

The generator proves playability rather than assuming it. For each level it:

- runs `parseLevel` to build the exact `blocked` set the game walks;
- runs the engine's own `findPath` for **every** spawn→exit and fails on no route;
- flags any spawn/exit that an obstacle sealed, any suspiciously short route, and
  any wave that names an undeclared spawn or creep type.

Because it reuses `src/core.js` (via the tiny Node DOM shim in
`tests/support/xml-dom.mjs`), a level that validates here behaves the same in the
browser. This is the single most important habit: **let the build reject a broken
map** instead of discovering it mid-play.

## Procedural Swarm

`src/procedural-levels.js` applies the same XML and validation pipeline at
runtime. An eight-digit hexadecimal seed, difficulty, economy mode, and generator
version completely determine a mission. The source name records all four values,
so continue/restart can reconstruct a mission even after its visible group is
rerolled. Fun adjective/noun names are derived from that identity as well; there
is no seed-to-name catalog to maintain.

Each difficulty and economy mode retains only its current four seeds. **Normal
Economy** fixes cash, lives, tower prices, and kill wealth at the documented
defaults. **Random Economy** varies them inside difficulty-specific safe ranges.
Both modes randomize entrances, exits, obstacles, special terrain, waves, creep
rosters, and the available non-Vortex towers.

Every candidate is parsed by `parseLevel` and rejected unless all spawn routes
exist, routes are at least eight cells long, entrances have room to branch, at
least 80 interior cells remain buildable, every spawn receives a wave, every wave
uses a declared creep, and the starting economy can buy five of its cheapest
tower. Generation uses bounded retries followed by a known-safe sparse fallback,
so malformed seeds cannot hang the page.

Procedural state retains independent **Standard** and **XL** sets. XL source names
include `_XL_`, use a 22×24 `<hexmap>`, require routes of at least fourteen cells
and at least 260 buildable interior cells, and can emit several `concurrent`
entries per apparent wave. Existing Standard v1 source names and seeded output
remain unchanged after the local-state migration.

## Adding a whole new campaign

A campaign is its own menu section with its own Easy/Medium/Hard tabs. The LHC
pack added one end-to-end; mirror those four edits:

1. **`src/level-index.js`** — add a source-name regex, return your tag from
   `classifyName`, and push into a new bucket in `buildLevels` (call
   `parseLevel(xml, sourceName, null, '<tag>')` and sort by `diffRank` then `id`).
2. **`src/main.js`** — import your `entries`, merge them in `applyLevels`
   (`buildLevels([...entries, ...yourEntries], parseLevel)`), toggle a menu button
   in `showMainMenu` (`$('menuX').hidden = !levels.some(l => l.campaign === '<tag>')`),
   and wire its click to `enterDifficulty('Easy','<tag>')`.
3. **`index.html`** — add the `<button id="menuX" class="action" hidden>…</button>`
   to `.native-menu-buttons`.
4. **Ship the data** — commit the generated `src/<name>-levels.js` and its
   generator script.

The campaign then flows through the same difficulty tabs, level cards, save/continue,
and local leaderboard as everything else — `renderLevels` filters purely on
`l.campaign` and `l.difficulty`.

## Fixed-path (classic) levels

Instead of a maze, provide a `<creepPath>` of `<point x="…" y="…"/>` vertices (in
the original iPhone 320×480 space; the engine maps it into the play field). The
presence of a 2-plus-point path flips the level into **path mode**:

```xml
<creepPath>
  <point x="0"   y="240"/>
  <point x="160" y="240"/>
  <point x="160" y="80"/>
  <point x="320" y="80"/>
</creepPath>
```

In path mode: creeps stream along the polyline, the hexes near the path become
unbuildable automatically, **Vortex becomes buildable**, and you don't author
`<spawnhex>`/`<exithex>` mazes. This is how the Original geoDefense campaign works;
see those levels (supply the `.ipa`) for full examples.

## Quick checklist

- [ ] Unique `file`/source name with the right `_E_`/`_M_`/`_H_` marker, and unique `id`.
- [ ] Spawn and exit on edge cells (or a reachable interior exit); every `<wave spawnHex>` names a real spawn.
- [ ] Every `<spawn type>` is declared in `<creeps>`.
- [ ] No `POP` in a Swarm level's `<towers>` (use `<placetower>` if you want a Vortex).
- [ ] Generator prints `✓` for the level, `npm test` passes, `npm run build` run.
